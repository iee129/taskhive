# 코드 리뷰 가이드

## 리뷰어 체크리스트

### 보안
- [ ] SQL 인젝션 위험 없음 (JPA 파라미터화 쿼리 사용)
- [ ] 민감 정보(비밀번호, 토큰)가 로그에 기록되지 않음
- [ ] 환경 변수로 주입해야 할 값이 하드코딩되어 있지 않음
- [ ] 인증이 필요한 엔드포인트에 `@Secured` 또는 Security 설정 적용됨

### 코드 품질
- [ ] 메서드가 단일 책임을 가짐 (50줄 이내 권장)
- [ ] 의미 있는 변수·메서드명 사용
- [ ] 중복 코드 없음
- [ ] 예외 처리가 적절함 (빈 catch 블록 금지)

### 테스트
- [ ] 새 로직에 대한 단위 테스트 존재
- [ ] 엣지 케이스 테스트 포함
- [ ] 테스트가 실제로 의미 있는 것을 검증함

### 문서
- [ ] 복잡한 로직에 주석 존재 (자명한 코드에 불필요한 주석 금지)
- [ ] API 변경 시 `docs/05-api/` 업데이트됨

## 피드백 작성 원칙

| 유형 | 접두사 | 예시 |
|------|--------|------|
| 필수 변경 | `[Required]` | `[Required] 비밀번호가 로그에 노출됩니다` |
| 제안 | `[Suggestion]` | `[Suggestion] 이 로직을 별도 메서드로 분리하면 어떨까요?` |
| 질문 | `[Question]` | `[Question] 이 케이스에서 null이 올 수 있나요?` |
| 칭찬 | `[Praise]` | `[Praise] 예외 처리가 깔끔하네요!` |

## 리뷰 응답 가이드

- `[Required]` 피드백은 모두 반영 후 resolve
- `[Suggestion]`은 반영 여부를 댓글로 설명
- 의견 불일치 시 오프라인 논의 후 결론 기록

## PR 크기 가이드

| 크기 | 변경 파일 수 | 권장 사항 |
|------|------------|---------|
| Small | 1~5 | 1일 내 리뷰 |
| Medium | 6~15 | 2일 내 리뷰 |
| Large | 16+ | 기능 단위로 분리 권장 |

큰 PR은 리뷰어의 부담이 높고 실수를 놓치기 쉬움 — 기능 단위로 작게 분리.
